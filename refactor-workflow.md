# AI 辅助模块重构的工程化工作流

让 AI 重构一个模块，最容易翻车的不是"它不会改"，而是两件事：**理解还没达标就动手**，以及**改着改着偏了，等你回头 review 才发现，回滚代价已经很大**。

重构的定义是：**行为不变地改结构**。既然行为必须不变，那"当前行为"就是整件事的锚点。下面这套工作流就是围绕这个锚点，把已有工具（`my-codebase-intel`、`planning-with-files-zh`、CodeGraph、superpowers）串成一条固定流程，核心三句话：**理解契约化、步骤可验证、改动原子化**。

## 工具串联总表

先看全貌，六个阶段各自用什么、产出什么：

| 阶段 | 用什么 | 产出 |
|------|------|------|
| ① 理解模块 | `my-codebase-intel` 模块介绍/重构评估 + CodeGraph `fn-impact`/`query`/`context` | `findings.md`（行为契约） |
| ② 理解达标 gate | 显式批准（brainstorming gate 思想） | 契约定稿 |
| ③ 写细计划 | `planning-with-files-zh` + writing-plans | `task_plan.md` + specs |
| ④ 验证门禁 | characterization tests / 脚本 / 手动复现 | 回归基线 |
| ⑤ 逐步执行 | git worktree + 每步原子 commit | `progress.md` 审计 |
| ⑥ 跑偏回滚 | 验证门禁早发现 + git revert/reset | 退到上一个好 commit |

## 阶段① 理解模块 → 产出"行为契约"

第一步和 AI 谈模块、谈代码关联，目的不是"让 AI 大概懂了"，而是产出一份**行为契约**。重构的成败由行为是否守住决定，所以这份文档真正要钉死的不是"这个模块大概做什么"，而是：

- **公开接口**：对外暴露的函数、类型、签名。
- **输入输出**：每个入口的参数、返回值、数据形态。
- **副作用**：写文件、发请求、改全局状态、触发事件。
- **不变量**：必须始终成立的约束（顺序、并发、幂等性等）。
- **当前所有调用方**：谁在用它，用的是哪些行为。

这些不靠 AI 凭空总结，靠查图。用 `my-codebase-intel` 的「模块介绍」「重构评估」配合 CodeGraph：

```text
mcpCodegraphContext   → 模块/函数完整上下文（源码 + 依赖 + 调用方）
mcpCodegraphQuery     → 拉出所有调用方（reverse=true 反向）
mcpCodegraphFnImpact  → 影响半径：改了会波及哪些函数
```

**关键观点：这份文档不是笔记，是行为契约 / 回归基线。** 它存进 `planning-with-files-zh` 的 **`findings.md`**——这个文件在 `/clear` 和上下文压缩后会自动恢复，正好跨越"理解阶段 → 执行阶段"之间最容易断裂的那道缝。等到第④阶段写验证，这份契约直接当回归检查清单复用。

## 阶段② 理解达标 gate（显式批准）

你提到"AI 的理解和我认为达标以后，才能让它重构"——这道 gate 很对，要把它**正式化**，而不是凭感觉。

做法：行为契约（`findings.md`）必须经你**显式批准**后，才允许进入写计划。批准前 AI 不碰代码、不写 specs。这正是 superpowers `brainstorming → writing-plans` 的 gate 思想：先对齐理解，再谈实现。一道明确的 gate 能挡掉"理解还差一截就开干"这个最贵的错误。

## 阶段③ 写细计划：plan-with-file + specs

理解定稿后，用 `planning-with-files-zh` 列计划：`task_plan.md` 排阶段顺序，specs 钉每一步。

这里有个**反直觉的粒度警告：specs 别细到逐行规定代码**。逐行 diff 看着"很细"，其实是陷阱——它锁死实现方式，还容易和真实代码脱节，AI 一旦发现现实和 spec 对不上就会卡住或硬凑。

最佳粒度是每一步钉住三件事：

```text
目标行为   —— 这步要达成什么（不是"怎么写代码"）
验收标准   —— 满足什么条件算这步完成
怎么验证   —— 用什么命令/测试/对比来确认
```

"怎么验证"才是后面抓"跑偏"的钩子，比逐行 diff 有用得多。

**步骤排序**：用 CodeGraph `mcpCodegraphFnImpact` 按影响半径排，**先改叶子、再往上改**。叶子节点调用方少、影响小，每步独立可验证、可单独提交。把影响面最大的核心节点留到最后，前面的步骤已经把它周围都验证过了。

## 阶段④ 验证门禁（本文核心）

这是回答你第三个问题"跑偏了怎么及时回滚"的关键，但视角要换一下：**重点不是"能不能撤销"，而是"多早发现跑偏"**。越早发现，回滚越便宜。git 能撤销是粗网，真正的细网是**每一步都可验证**。

立一条原则：**每个重构步骤必须可验证，验证通过才进下一步。** 具体手段按项目情况分级降级：

| 项目情况 | 验证手段 |
|---------|---------|
| 有测试框架 | 重构前先写 **characterization tests**（钉住当前行为的测试），改完跑测试，绿了才算行为没变 |
| 无测试框架 | 降级到脚本复现 / 手动复现关键路径 / 关键输入输出对比 |

characterization test（也叫 golden master）的思路：在动手前，给模块当前行为拍一张"快照"——用真实输入跑一遍，把输出固化成断言。重构后这些断言还全绿，就证明行为守住了。这和 superpowers `test-driven-development` 同源。

注意降级路径不是"没测试就跳过验证"。没有测试框架时，可以写个临时脚本喂固定输入、比对重构前后的输出；或者手动跑通关键路径、记下结果。**只要能在每步之后回答"行为变了没"，门禁就成立。**

## 阶段⑤ 逐步执行 + 原子化改动

执行时两条纪律：

1. **隔离环境**：在独立分支或 **git worktree** 上做，main 保持干净。worktree 的好处是整个重构环境物理隔离，真要废弃时直接丢掉整个目录，不污染主工作区（参见 superpowers `using-git-worktrees`）。

2. **原子 commit**：每完成一个 spec 步骤 = 一个 commit。commit message 写清这步守住了什么行为、怎么验证的。这样每个 commit 都是一个"已知良好状态"。

同时让 `progress.md` 每步记三件事：

```text
改前状态：这步动手前模块/测试是什么样
改后状态：改了哪些文件、哪些函数
验证结果：门禁过了没、用什么验的
```

`progress.md` 是审计轨迹。一旦后面发现行为偏了，回看它能一眼定位是从哪一步开始发散的。

## 阶段⑥ 跑偏了怎么办

按"早发现 → 能回滚"两层应对：

- **第一道防线是验证门禁红了**。某步改完测试不绿、或输出对比对不上，当场就停——这时候只偏了一步，根本不用"回滚"，改回来就行。
- **真要回滚**：`git revert` 或 `reset` 退到上一个原子 commit（也就是上一个已知良好状态）。因为每步都是独立 commit，回滚精确到步，不会牵连其他改动。
- **worktree 模式下更干脆**：整个隔离环境可以直接丢弃，main 从头到尾没被碰过。
- **定位发散点**：用 `progress.md` 对照，找出"验证结果"第一次出问题的那一步，根因往往就在那。

## 小结

这套流程没有引入任何新工具，只是把已有的串成闭环：

- 你的问题一（理解存哪）→ **`findings.md` 当行为契约**，跨越上下文断层。
- 你的问题二（计划要细）→ **specs 钉"目标行为 + 验收标准 + 怎么验证"**，而不是逐行代码。
- 你的问题三（跑偏回滚）→ **验证门禁是细网（早发现），git 原子 commit 是粗网（能撤销）**。

贯穿始终的一条线：重构守的是**行为**，所以从理解到执行，每一环都围绕"当前行为是什么、有没有变"展开。
