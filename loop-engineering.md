# /loop 与循环工程实践

让 AI 自己跑一个长任务,最容易翻车的不是"它不会做",而是两件事:**循环停不下来**(没有明确终止条件,白烧 token),以及**每一轮都在重新开始**(状态没落盘,上下文一压缩就丢)。

所谓"循环工程"(Loop Engineering),本质就是把一个目标拆成三个要素:**可重入的循环 + 明确的终止条件 + 持久化的状态**。三者缺一不可——少了终止条件就死循环,少了持久化状态就每轮失忆,少了可重入设计就重复劳动。`/loop` 是 Claude Code 提供的循环原语,下面先讲它的三种模式,再讲围绕这三要素的三种落地范式。

## `/loop` 的三种模式

`/loop` 是 Claude Code 的**内置 skill**(不在插件文件里,由 harness 直接提供)。它把一个提示词或 slash command 按节奏反复执行,核心用途是"无人值守地推进一个长任务"。给不给时间间隔,决定了它走哪种模式:

| 模式 | 触发方式 | 底层机制 | 适用场景 |
|------|---------|---------|---------|
| **Interval(定时)** | 带时间间隔 | cron:同一 prompt 固定 wall-clock 间隔重新入队 | 轮询外部状态(CI、部署、PR 队列) |
| **动态(自定节奏)** | 不带时间间隔 | 模型用 `ScheduleWakeup` 自己决定下次何时醒来 | 任务节奏不固定,等什么就睡多久 |
| **裸跑** | 什么都不带 | 跑内置"维护提示词",检查上下文、推进未完成工作 | 让 Claude 自动接着干 |

```text
/loop 5m /babysit-prs           # interval:每 5 分钟跑一个 slash command
/loop 10m 检查部署状态,失败就回滚   # interval:每 10 分钟跑一段自然语言任务(支持 s/m/h/d)
/loop 持续重构这个模块直到测试全通过   # 动态:不给间隔,模型自选节奏
/loop                            # 裸跑:内置维护提示
```

**动态模式的节奏怎么选**:模型会根据"在等什么"来决定延迟。等 CI 就压在 270 秒内(保持提示缓存热),纯空转就 20-30 分钟一次。不要盲目设 `/loop 1m`——高频空转既烧 token 又不断打断缓存。

## 三种落地范式

### 方案 A — Ralph Loop:同一提示反复喂回(适合开放式攻坚)

官方 `ralph-loop` 插件实现了 "Ralph Wiggum" 技术,一句话概括就是 **"Ralph is a Bash loop"**:一个 `while true` 反复把同一个 prompt 喂给 AI,直到完成。它不靠外部 bash 循环,而是用一个 **Stop hook** 在当前 session 内拦截 Claude 的退出尝试:

```text
你运行一次:
/ralph-loop "实现完整的 OAuth 流程并通过所有测试" --completion-promise "所有 auth 测试通过" --max-iterations 20

然后 Claude Code 自动:
1. 做任务  2. 尝试退出  3. Stop hook 拦截退出
4. Stop hook 把同一 prompt 喂回  5. 重复直到完成
```

关键设计:

- **prompt 永不改变**,每轮 Claude 看到的是上一轮留在**文件和 git history** 里的成果,靠读自己的过往工作持续迭代——所以状态全靠落盘,不靠对话记忆。
- **`--completion-promise`**:只有当这句话"完全、毫无疑义地为真"时,才允许输出它来跳出循环。这道约束防止模型觉得卡住了就输出假承诺提前逃跑。注意它是**精确字符串匹配**,无法表达多种结束条件(比如 "SUCCESS" vs "BLOCKED")。
- **`--max-iterations`**:硬上限,是防失控的**首要兜底机制**。即使有 completion-promise,也务必设它,防止在不可能完成的任务上无限循环。

适合:有明确成功标准、需要反复迭代(如把测试跑绿)、有自动验证手段(测试/linter)、可以走开不盯的活儿。不适合:需要人做设计判断、一次性操作、成功标准模糊、生产环境调试。

### 方案 B — Plan-Loop:计划驱动的看护(适合有明确阶段的工程)

如果装了 `planning-with-files` 插件,它提供 `/plan-loop`——一个 plan 感知的 `/loop` 封装:

```text
/plan          # 先生成 task_plan.md / progress.md / findings.md
/plan-loop 10m # 每 10 分钟一个 tick
```

每个 tick 默认做的事是固定的:重读 `task_plan.md` 和 `progress.md` → 跑 `check-complete.sh` 看剩余阶段 → 自上次 tick 没有新进展就写一条 progress 摘要 → 有阶段完成就更新它的 `Status:` 行 → 还有活就推进下一阶段。

比裸 `/loop` 强的地方:**每个 tick 都强制重新落地到 plan 文件**,而不是靠对话记忆——这正好跨越了"理解阶段 → 执行阶段"之间最容易断裂的那道缝。配合 `/plan-goal` 设终止条件,plan 完成时循环自动停。这是"替我看着这个计划做完"的标准范式。

### 方案 C — interval 轮询:等外部信号(适合 CI / 部署 / PR 队列)

```text
/loop 5m 检查 gh pr checks,全绿就合并并通知我,有失败就贴出失败的 job
```

**一个重要反模式**:如果只是"等某件事完成、发一次通知就够了",**不要用 `/loop`**——那该用后台命令配一次性通知,否则循环会一直空转。`/loop` 只适合"每次发生都要处理"的**持续轮询**。

## 工程纪律

把上面三种范式提炼成五条通用纪律,做 Loop Engineer 时对照检查:

| 要素 | 做法 | 反模式 |
|------|------|--------|
| **状态持久化** | 每轮成果落盘到文件/git,开头先读现状 | 依赖对话记忆(会被 compaction 清掉) |
| **终止条件** | completion-promise / plan-goal / max-iterations 至少选一 | 无界循环,靠人盯着喊停 |
| **节奏选择** | 等 CI 用动态模式;固定轮询用 interval | 盲目 `/loop 1m`,烧 token 又打断缓存 |
| **幂等性** | 提示词写成"检查 → 若未完成则推进" | "再做一遍 X",导致重复劳动 |
| **可观测** | 每轮写一条 progress 摘要 | 黑盒跑完,不知道中间发生了什么 |

## 选型一句话

> 开放式攻坚选 **Ralph Loop**,有阶段计划选 **Plan-Loop**,等外部状态选 **interval 轮询**,节奏不定选**动态模式**。

最后给一个可直接复制改用的 Ralph 提示词模板,把"成功标准 + 自纠正 + 逃生舱"都写进去:

```text
/ralph-loop "实现 <功能>,遵循 TDD:
1. 先写失败的测试
2. 实现功能
3. 跑测试
4. 有失败就调试修复
5. 必要时重构
6. 重复直到全绿
完成时输出:<promise>COMPLETE</promise>
若到第 15 轮仍未完成:记录卡点、列出已尝试的方案、给出替代思路。" \
  --completion-promise "COMPLETE" --max-iterations 20
```
